1 Motivation

1.1 Need for a human readable semi-structured markup language

If we define a human readable semi-structured markup language as follow :

Human readable : It uses more letters and digits than other characters because they are easier to read. When it uses non alphanumerical characters, it will be those easy to type : equal, slash and backslah, plus, dash, star, sharp, period.

Semi-structured : The two category of uses are schema and data containment.

Markup : Markups can be spreaded like a template, or be structured in a hierarchical fashion. Because use case are beyond HTML/XML needs (configurations, data transmissions, etc), the hierarchical style has been choosen while remain template capable.

Language : A language, if we want it to be used, it should follow the evolution of the use cases. It should not be hard to evolve. It can be as extensible as XML for example.

A "lot" of semi-structured markup language exist (see Lists or surveys of markup languages). XML dominates all by its completeness (see XML benefits) but not its simplicity (see XML drawbacks).

2 This document format

2.1 3ML Syntax format

  • The elements (non leaves) are Backus-Naur broken until leaves.
  • "{ }" are used to designate a vertex of the Backus-Naur tree.
  • "( )" are used to designate a leaf of the Backus-Naur tree.
      Leaves are mapped to characters, string or sets in the Proposed and Voted Special Characters and Character Sets table.
  • "a..b" are used to express cardinality.
      1..1 or 1 is default thus omitted.

2.1.1 Example:

{Header}(Header-value Separator){content 0..1}

  Thus "0..1" means here an optional element

3 Conformance

This specification use the conformance key words from this Request For Comments : RFC 2119, except shall not, as validation qualifiers.

These key words are : Required, Recommended, Not Recommended and Avoid (more clear than shall not)

These conformance key words are transferred from the section's pre-requirement section. If inferred or created they must be supported by referenced argument[s].

These key words are used to group listed requirements (in tables) in priority order. Their meaning may be contextualized in their respective section.

All exemples are not normative.

The words Used and Inspired, qualify each source of arguments.

4.2 The 3ML markup language

4.2.1 Is

3ML is a declarative markup language mostly inspired by the XML.

Other inspirations are :

  • JSON
    • Wikpedia
  • JSON Schema
    • Wikpedia

Anti-inspirations are :

  • YAML
      Lack of modular pattern that may hinder evolutivity.
    • Wikpedia
  • Markdown : Same as YAML
    • Wikpedia

basically structured in a parent-child fashion except for the physical container which has two sequential sections : the header and the content which in their turn has parent-child structure as content. Some features such as templating are structured in a placeholder fashion instead.

The physical containers can be file, databases, variables, etc. The logical containers are nodes. Content can be node or text. Embedded binairies are not supported yet but can be described or pointed to.

3ML main focus and corner stones are the nodes. Even the physical containers can be designated as nodes.

4.2.2 Is not

3ML is not a procedural but may describe a process using sibling nodes sequence as steps.
